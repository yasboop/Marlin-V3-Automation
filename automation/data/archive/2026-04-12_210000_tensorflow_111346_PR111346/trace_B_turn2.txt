‚ĚĮ The processSwitch method in RegionMaker is taking a surprisingly large
share of total decompilation time. Profiling shows the bottleneck is in
post-dominator computation - right now BlockUtils.calcPostDominance and
calcPartialPostDominance are getting called repeatedly during switch case
processing, each time building a fresh HashMap from scratch for almost every
 case in every switch. Apps that went through Redex optimization tend to
have lots of large switches which makes this really noticeable.

The dominator tree is already computed once per method and cached on
BlockNode (the doms and idom fields). Post-dominators should work the same
way - compute them once during block processing, store them on the block
nodes, and then RegionMaker can just read the cached info instead of
recomputing.

The existing DominatorTree uses an iterative algorithm that should be
reusable for post-dominators if you reverse the graph traversal direction.
Post-dominator computation walks backwards from the exit block and uses
successors where the forward computation uses predecessors.

After caching post-dom info on block nodes, the switch out-block search and
fallthrough detection in RegionMaker needs to be updated to use it instead
of the old on-demand calculations. The old utility methods in BlockUtils for
 partial post-dominance wont be needed anymore, those should go. The DFS
traversal in BlockUtils needs to support reverse direction for the backwards
 walk. All existing tests including the switch decompilation tests should
pass without issues.
