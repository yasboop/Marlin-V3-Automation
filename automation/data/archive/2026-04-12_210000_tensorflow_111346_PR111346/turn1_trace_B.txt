  Summary of Changes

  Problem

  XLA's Memory Space Assignment algorithm was too aggressive about evicting
  conditional operands from alternate memory. When a buffer in VMEM was passed
  into a conditional, the algorithm forced it back to default memory even when
  all uses inside the branches could read from alternate memory.

  Solution

  7 files changed, 286 insertions(+), 3 deletions(-)

  1. options.h: Added enable_conditional_redundant_eviction_elimination flag
  (default false for backward compatibility).
  2. allocation.h / allocation.cc:
    - Extended MirroredAllocation with a new constructor accepting an explicit
  MemorySpace parameter, enabling mirrored allocations in alternate memory
  (previously hardcoded to default memory).
    - Updated MirroredAllocation::Process() to be a no-op for alternate memory
  mirrored allocations (no instruction rewriting needed since the buffer is
  already in place).
    - Improved MirroredAllocation::ToString() to include memory space and chunk
  info for debugging.
  3. algorithm.h / algorithm.cc:
    - Added MaybeCreateMirroredAllocationForConditionalUse() method that checks
  if all branch uses can work with alternate memory, and if so, creates mirrored
   allocations in VMEM for branch parameters pointing to the same chunk.
    - Modified the eviction logic in CreateAllocationRequest() to skip forced
  eviction when mirrored alternate-memory allocations exist for the conditional.
    - Added conditionals_with_mirrored_alternate_mem_ tracking set, cleared
  properly in ClearPendingChunks().
    - Integrated the new method into AllocateAllocationValues alongside the
  existing while-loop mirrored allocation logic.
  4. memory_space_assignment.cc: Updated VerifyAllocations() to skip
  MirroredAllocations in alternate memory from overlap checking, since they
  represent references to the same physical chunk as the original allocation.
  5. memory_space_assignment_test.cc: Added two new tests:
    - Happy path: Verifies that when all branch uses are short, the buffer stays
   in alternate memory and the subsequent use can access it without re-prefetch.
    - Fallback: Verifies that when a branch has uses too far from the parameter
  (long negate chain), the optimization falls back to eviction.
